# 有效工时

> 新成立的"技术管理中心"把浩宇写了 2.5 小时的三版方案判定为"零有效工时"——因为审核未通过期间的工作"属于个人自驱行为，不等于产出"。线上 P0 支付故障 2000 用户投诉，紧急解锁需要 VP 和 CTO 双签，但 VP 在见客户、CTO 在出差。一周后 VP 看到"IT 部门有效工时仅 18%"的数据，表扬了技管中心，然后发现浩宇已经离职了。

## 技术真相

公司从 IT 部门抽调人员成立"技术管理中心"，职责是"帮大家提效"。实际运行的是一套层层嵌套的审批体系：需求先过技术审核才能立项（1-3 个工作日），立项后才能做调研写方案，方案需要密等性、灰度回滚等完整文档（被以前的测试同事逐项挑刺），方案审核通过后还有排期审核（等下午 4 点排期会），排期审核后系统同步需要 10-30 分钟。在此期间，开发时间不计入"有效工时"——因为审核未通过的工作属于"个人自驱行为"。

紧急情况（P0 支付故障）的解锁需要 VP 和 CTO 双签电子授权，但两者均不可达。技管中心拒绝绕过流程："如果我们因为着急就绕过流程，那流程还有什么意义？"

最终数据：IT 部门有效工时仅 18%，VP 认为"八成时间在做无效的事"，表扬了技管中心的量化体系。

## 领导话术

- 公司鼓励自驱，但自驱不等于产出——需求先过审核才能立项，立项前的工作属于个人行为
- 不是不算，是暂时不计入，性质不一样的
- 如果我们因为着急就绕过流程，那流程还有什么意义？
- 数据不会说谎，IT 部门有效工时只有 18%，过去我们以为他们在忙，其实八成时间在做无效的事
- 如果不是技管中心建立了量化体系，我们永远发现不了
