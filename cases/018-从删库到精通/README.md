# 从删库到精通

> 实习生前端出身，登录生产库 root 跑 SQL 删了订单表，又手滑清了 binlog、关了消息队列，三道防线全破。老程序员浩宇发现凌晨全量备份还在，消息队列虽然被关但消息堆积未丢，用"备份恢复 + 消息重播"一把找回全部增量数据，三十分钟搞定。对外只说"系统升级"，对内一句"不确定不按回车"。

## 技术真相

实习生以为生产库是测试库（终端窗口看着一样），在生产库 root 上执行了 DELETE，把订单表全删了。然后执行了 `PURGE BINARY LOGS`，以为是恢复命令，实际上是清除 binlog——增量恢复的第一道防线没了。又顺手关了消息队列——增量恢复的第二道防线也关了。从库延迟两小时，上周就报了没人修——第三道防线也废了。

唯一的救命稻草：凌晨三点的定时全量备份还在。消息队列虽然被关了，但消息没有丢，全部堆积在队列里。恢复方案：先用凌晨全量备份恢复数据库到三点的状态，再把消息队列里堆积的操作消息重播一遍，补回三点到现在的全部增量。数据一条没丢，包括客户那三笔。

对外口径：系统升级中，数据同步延迟，三十分钟恢复。不写故障，写升级。

## 领导话术

- 不是删库，是系统架构升级过程中的一次数据同步操作
- 消息队列具备天然的数据堆积与重播能力，这是我们架构设计的核心优势
- 全量备份 + 消息重播，数据零丢失，三十分钟内恢复，远超行业平均水平
- 传统方案依赖 binlog 做增量恢复，我们的方案基于消息队列重播，更可靠、更可控
- 这次事件验证了我们消息驱动架构的容灾能力，建议纳入常态化演练体系
- 三十分钟就是三十分钟，说出去的话收得回来
