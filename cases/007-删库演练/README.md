# 删库演练

> 浩宇写清日志脚本时 `pass` 变量没赋值，`rm -rf` 直接干掉了整台生产服务器，备份也在同一台机器上一起没了。旭阳在王处来验收前两小时内，把"删库事故"包装成"数据安全实战化无预警应急演练"，还规划出一年四季度递进式演练体系，直接变成年终总结亮点。最后数据靠九如两年前随手配的、自己都忘了的 cron 定时备份任务救回来。

## 技术真相

浩宇写了一个清日志脚本，执行 `rm -rf "$pass"` 时 `pass` 变量还没赋值，展开成了 `rm -rf /`，把整台生产服务器的文件全删了。本地备份（beta backup）在同一台机器上，一并被清掉。异地备份没有——王处没批那个预算。App 还能打开，但所有接口都报错，王处两小时后就要来验收移动端适配。

实际救回来的是九如两年前随手配的一个 cron task，每周六凌晨把数据库 diff 到 NAS，全量 dump 2.3GB。这个任务连九如自己都忘了，但一直在跑，最后一次备份是昨天凌晨。最终数据库全量恢复，附件丢了，但王处从没点开过附件，只看列表数字。

## 领导话术

- 不是删库，是无预警触发——真正的灾难不会提前通知，所以真正的演练也不能提前通知
- 从数据归零到全量恢复，85分钟，系统自动触发容灾链路，异地备份拉起，完整回灌
- 别人写自查报告，我们真删库真恢复；别人是纸上谈兵，我们是真枪实弹
- 全市首家数据安全实战化演练，上面发通知"鼓励有条件的单位开展应急演练"，全市零执行，我们做了
- 一年四季度递进式演练体系：Q1 数据归零测恢复、Q2 模拟网络攻击、Q3 物理断电测冷启动、Q4 复合灾难全面压测
- 这不叫修 bug，叫体系建设——数据安全常态化演练体系
